## 第一部分：数据真实性审计（核心问题）

### 1.1 数据资产全景

|---------|------|--------|--------|----------|

| 专业真实录取线 | 声称来自第三方爬取 | ⚠️ **无法验证** | **66% (2024), 74% (2023)** | 与"录取最低分"列**100%矛盾** |

### 1.2 数据质量核心发现

**发现一："录取最低分"列 100% 为空——这是最大的数据谎言**

CSV 中明确有一列叫 `录取最低分`（官方标准命名），该列**全部为空**。这意味着：
- 项目声称拥有"专业真实录取线"（A 级数据置信度）
- 实际上，代码中把另一列叫 `最低分`（66% 填充）的数据当作"专业真实线"
- 但 `最低分` 与 `录取最低分` 是**两列不同的数据**，且 `最低分` 的数据来源不明

**这意味着：项目数据库中大量标记为 A 级置信度的数据，可能是错误标注的组线数据或第三方不可验证的爬取数据。**

**发现二：多组可疑的相同分数位次组合**

在 2023 年数据（`最低分.1`/`最低位次.1`）中：
- 清华大学 04 组社会科学试验班：676 分 / 101 位次
- 上海交通大学 03 组电子信息类：676 分 / 101 位次

两所不同顶尖高校的不同专业，出现完全相同的分数和位次，**这在真实录取数据中概率极低**。强烈暗示数据合并或爬取过程中存在错误。

**发现三：CSV 列名混乱，数据血缘不可追溯**

```
专业组最低分 / 专业组最低位次          → 2024 年组线（官方，已验证）
专业组最低分.1 / 专业组最低位次.1      → 2023 年组线（来源不明）
最低分 / 最低位次                      → 声称 2024 年专业线（来源不明）
最低分.1 / 最低位次.1                  → 声称 2023 年专业线（来源不明）
最低分.2 / 最低位次.2                  → 声称 2022 年专业线（仅 41% 填充）
专业组最低分_2025 / ...                → 2025 年组线（93% 填充，但 2025 年录取尚未结束）
```

**问题：2025 年录取尚未结束（当前为 2025 年 6 月），何来 93% 的 2025 年组线数据？**

经检查代码注释：`"2025 来自湖北省教育考试院官方 PDF"`。但湖北省教育考试院在 2025 年 6 月尚未发布本科普通批投档线（通常在 7 月中下旬发布）。因此这些 "2025 年数据" 极可能是：
- 2025 年模拟填报/适应性演练数据，**非真实录取数据**
- 或者代码中的 `LATEST_HISTORICAL_YEAR = 2025` 是一个前瞻设定，但数据实际缺失

**发现四：院校标签是主观评级，非官方分类**

CSV 中的 `院校水平` 列包含："世界一流大学"、"区域知名大学"、"中国一流大学" 等。这些标签来自第三方（如校友会、软科排名），**并非教育部官方分类**。但在推荐排序中，这些标签可能被隐含使用（虽然代码中声称已移除硬编码权重）。

### 1.3 官方数据验证结果

通过 Web 搜索获取的湖北省教育考试院 2024 年官方数据：

| 学校 | 专业组 | 官方投档线 | 官方位次 | CSV 组线 | CSV 位次 | 匹配结果 |
|------|--------|-----------|---------|---------|---------|----------|
| 清华大学 | 04 组 | 692 | 71 | 692.0 | 71.0 | ✅ 完全匹配 |
| 清华大学 | 06 组 | 690 | 98 | 690.0 | 98.0 | ✅ 完全匹配 |

## 第二部分：推荐算法真实测试与评估

### 2.1 概率估算公式测试

```python
def estimate_probability(candidate_rank, ref_rank):
    rank_ratio = candidate_rank / ref_rank
    prob = 1.0 / (1.0 + max(0, rank_ratio - 0.85) * 4.0)
    return max(0.02, min(0.99, prob))
```

**测试场景（假设考生位次 25000）：**

| 场景 | 考生位次 | 参考位次 | 概率 | 档位 | 问题 |
|------|---------|---------|------|------|------|
| 同位次 | 25000 | 25000 | 62.5% | 稳 | 同位次概率不到 100%，直觉合理？ |
| 位次优于 15% | 25000 | 21250 | 100% | 保 | 仅 15% 优势就 100%？过于乐观 |
| 位次差 30% | 25000 | 32500 | 38.5% | 冲 | 30% 劣势就 38.5%，可能过于乐观 |
| 位次差 100% | 25000 | 50000 | 17.9% | 冲 | 差距巨大，17.9% 是否合理？ |
| 高分段同位次 | 100 | 100 | 62.5% | 稳 | 高分段同位次概率与低分段相同？ |
| 高分段差 50% | 100 | 150 | 62.5% | 稳 | 前 100 vs 前 150，概率不变？ |

**核心缺陷：**

1. **公式是纯粹的数学构造，没有任何历史录取统计支撑**。一个负责任的高考志愿系统，概率应该基于：过去 3-5 年该专业录取位次的均值、方差、趋势，以及当年招生计划变化。
2. **参数 `plan_count` 和 `trend` 被声明但完全忽略**，这是代码层面的欺骗——看起来考虑了招生计划和趋势，实际上没有。
3. **不同分数段没有区分度**。高分段（前 1000 名）位次波动远小于中低分段（5-10 万名），但公式一视同仁。
4. **在 ratio = 0.85 处存在突变**：从 100% 直接跳到 62.5%，缺乏平滑过渡。

### 2.2 等位分换算测试

```python
def rank_to_equivalent(candidate_rank, from_year, to_year, province, subject_type, db):
    equivalent_score = _score_for_rank_cached(rank_cache, from_year, candidate_rank)
    to_rank = _rank_for_score_cached(rank_cache, to_year, equivalent_score)
    return to_rank
```

**逻辑正确，但依赖前提致命：**
1. **需要完整的一分一段表覆盖所有分数段**。如果 `rank_cache` 缺失某些年份或某些分数段，会直接报错（`ValueError`）。
2. **假设两年间分数分布线性可转换**。实际上高考难度、考生人数、批次线每年变化，简单的位次对应分数转换会引入误差。
3. **湖北省 2024-2025 年物理类分数大约下降 7-8 分**（同位次对比），简单的等位分转换没有考虑这种系统性偏移。

### 2.3 冲稳保分档逻辑测试

代码中 `risk_pref == "balanced"` 的分档：
- 冲：概率 15% ~ 45% → 对应 `ref_rank` 在 21578 ~ 29412（假设考生位次 25000）
- 稳：概率 45% ~ 75% → 对应 `ref_rank` 在 21578 ~ 29412
- 保：概率 >= 75% → 对应 `ref_rank` >= 29412

**问题：**
1. 分档仅依赖概率，**没有考虑专业组内专业数量、热门程度、调剂风险**。
2. 代码中 `max_per_school = 2`，同一学校最多推荐 2 个组，这**与湖北 45 个平行志愿规则不冲突，但过于保守**——湖北考生完全可以把同一学校的所有意向组都填上。
3. 排序函数 `_sort_key = group_prob * 10 + avg_relevance * 5` 中，**概率权重是相关度的 2 倍**。这意味着一个概率稍高但完全不相关的专业组，会排在一个概率稍低但高度相关的专业组前面。这是不合理的——考生应该优先看到与其意向最匹配的专业组。

### 2.4 LLM 在决策链中的过度依赖

整个推荐流程中 LLM 被调用了 3 次：
1. **画像解析**：`parse_profile_with_llm` — 必须调用，否则报错
2. **专业匹配**：`build_major_intent` + `score_major_relevance` — 必须调用
3. **最终决策**：`generate_rationale_tool` — 从候选池中重新排序和筛选

**问题：**
1. **画像解析没有 API Key 就报错**，系统完全无法降级运行。这在一个严肃的高考志愿系统中是不可接受的——应该有完善的规则兜底。
2. **最终决策把 LLM 当作"决策者"**，让 LLM 从候选池中选择索引。这引入了 LLM 幻觉风险：LLM 可能选择不存在的索引，或者完全忽略考生的关键约束。
3. **LLM 没有访问实时数据库的能力**，它只能基于文本摘要做判断，无法精确计算概率。

## 第三部分：工程与架构问题

### 3.1 数据层：无数据血缘、无校验、无版本控制

| 问题 | 严重程度 | 说明 |
|------|---------|------|
| 无数据血缘追踪 | 🔴 致命 | 不知道每条数据的原始来源、爬取时间、处理版本 |
| 无数据校验层 | 🔴 致命 | 数据导入时只有简单的 `_parse_int`，没有异常值检测 |
| 无数据版本控制 | 🔴 致命 | 无法回滚到上一版数据，无法 A/B 测试数据质量影响 |
| 无回测框架 | 🔴 致命 | 无法验证"如果去年用这个系统推荐，准确率是多少" |
| SQLite 单文件 | 🟡 严重 | 生产环境无法横向扩展，无并发安全保障 |

### 3.2 代码层：测试缺失、错误处理不足

- **45 个 Python 文件，0 个测试文件**（无 `tests/` 目录）
- **错误处理**：画像解析失败直接报错，没有优雅降级
- **类型安全**：大量使用 `Optional[str]` 和 `Any`，运行时容易出错
- **缓存问题**：`_schools_cache` 是全局进程内存，重启后失效，无法分布式部署

### 3.3 算法层：概率公式未经校准

这是最关键的算法缺陷。一个负责任的高考志愿系统应该：
1. 收集过去 N 年的录取数据
2. 计算每个专业组/专业的录取位次分布（均值、标准差）
3. 基于考生位次与历史分布的相对位置，计算录取概率（可用正态分布或经验分布）
4. 考虑招生计划变化做修正

当前系统的概率公式是：**一个人坐在电脑前拍脑袋想出来的**，没有任何统计基础。

### 4.1 对考生的潜在伤害

高考志愿填报是**人生关键决策**。当前系统存在以下风险：

| 风险场景 | 后果 | 当前系统是否避免 |
|---------|------|---------------|
| 用组线替代专业线，低估热门专业难度 | 考生高估录取概率，填报后滑档 | ❌ 未避免，系统大量推荐基于组线 |
| 概率公式过于乐观 | 考生"冲"的志愿太多，"稳"和"保"不够 | ❌ 未避免，同位次概率 62.5% 偏高 |
| 数据错误（如可疑的 676/101） | 推荐基于错误数据的院校 | ❌ 未避免，无数据校验 |
| LLM 幻觉导致推荐不匹配专业 | 浪费志愿名额 | ❌ 未避免，LLM 做最终决策 |
| 缺少官方数据标注 | 用户无法判断哪些数据可信 | ❌ 未避免，报告无数据来源说明 |

### 4.2 免责声明的不充分

当前 `README.md` 中的免责声明：

**问题：**
1. 仅在前端展示是不够的，**每个推荐结果都应该附带数据来源和置信度说明**。

3. 没有告知用户：LLM 生成的推荐理由可能包含不准确的信息。

## 第五部分：全局批判性建议（按优先级排序）

### 🔴 P0：数据真实性（立即执行）

1. **暂停使用所有标记为 A 级置信度的数据**
   - 当前 "A 级"（专业真实线）数据不可信，因为 `录取最低分` 列全空，而 `最低分` 列来源不明
   - 将所有数据降级为 C 级（组线估算），直到获得可验证的专业线数据

2. **建立数据血缘追踪系统**
   - 每条 `MajorScore` 记录必须标注：原始来源 URL、爬取时间、处理版本、校验状态
   - 删除所有无法追溯来源的数据

3. **从官方渠道获取数据**
   - 组线：直接从各省教育考试院 PDF/Excel 下载
   - 一分一段表：从省教育考试院官方发布获取
   - 专业线：从各高校本科招生网获取（唯一可信来源），或与高校招生办合作

4. **建立数据校验流水线**
   - 组线数据必须与官方 PDF 逐项比对（可用 OCR + 规则匹配）
   - 异常值检测：分数与位次是否匹配当年一分一段表？同一专业组内专业线是否都 >= 组线？
   - 交叉验证：同一年份、同一学校、同一组的多来源数据是否一致？

### 🔴 P1：算法重构（2-4 周）

1. **概率公式必须基于统计校准**
   ```python
   # 建议方向：基于历史位次的经验分布
   def estimate_probability(candidate_rank, historical_ranks: List[int], plan_trend: float):
       """基于历史 3-5 年录取位次的均值和标准差，计算考生位次的录取概率。"""
       if not historical_ranks:
           return None  # 明确返回未知，而不是猜测
       mean_rank = np.mean(historical_ranks)
       std_rank = np.std(historical_ranks) or 1.0
       # 使用正态分布或经验分布计算 P(录取位次 >= candidate_rank)
       # 考虑招生计划趋势修正
       adjusted_mean = mean_rank * (1 + plan_trend * 0.1)  # 招生增加 10% → 位次放宽 1%
       z_score = (candidate_rank - adjusted_mean) / std_rank
       # 当 candidate_rank < adjusted_mean（位次更靠前），概率更高
       prob = 1 - norm.cdf(z_score)
       return max(0.02, min(0.99, prob))
   ```

2. **引入回测框架**
   - 用 2022-2024 年数据模拟：如果 2023 年考生使用本系统，推荐结果的"录取率"是多少？
   - 定义指标：Top-1 命中率、Top-10 命中率、滑档率、冲刺成功率
   - 如果回测结果不理想，必须调整算法

3. **分档逻辑考虑专业热度**
   - 同一个专业组内，热门专业（计算机、临床医学）的实际录取位次可能比组线高 5000-20000 名
   - 建议引入"组内偏移量"：基于专业历史录取数据与组线的差距，调整推荐概率

## 总结

Spiral 项目在技术架构上有良好的设计（FastAPI + React + Agent 模式），但其**数据层存在严重的真实性和完整性问题**，**算法层缺乏统计基础**，**伦理层对考生风险的保障不足**。

**当前项目状态：不能用于指导真实考生填报志愿。**

1. 专业级录取数据不可信（A 级数据注水）
2. 概率公式未经校准，可能系统性高估录取概率
3. 用组线替代专业线，对热门专业的录取难度系统性低估
4. 缺乏数据校验和回测，无法保证推荐质量

**建议立即行动：**
1. 冻结所有 A 级数据，降级为 C 级

3. 重写概率估算公式，基于历史统计校准
4. 引入回测框架，量化系统准确率
5. 在推荐结果中增加详细的数据来源和风险提示

只有完成以上改造后，该系统才能考虑对外提供志愿填报服务。
